Ship the Summarize button — thin, on-demand, cached by version.

Three independent reviews of the "AI summary per hosted site" idea converged on the same core architecture. Build it as a lazy button, not an upload hook. The disagreements were about plumbing — adjudicated below.

01Verdict

Build — with a narrow scope

Worth building as an on-demand feature: a button that summarizes the site's HTML via Azure OpenAI when configured (Workers AI fallback for keyless forks), stores the result against the site's contentVersion, and regenerates only when the content — or the summarization prompt — actually changes (a quiet forced escape exists; see the round-2 ruling). Workers AI is already live in prod (voice-comment transcription), so this is one migration, two pure helpers, one route, and one sheet component — no new infrastructure.

Honest caveat (both external voices raised it): value is narrower than it sounds. Chart/JS-heavy dashboards extract poorly server-side (JS never renders) and audio sites have nothing to summarize. It shines for text-heavy explainers — still a large slice of what Glance hosts. Cost is a non-issue when cached.

codex · gpt-5.6 xhigh

"Narrow but real value — helps reviewers triage before opening. Viable only if strictly on-demand." Priced it: ~102 neurons per summary on a 3B model.

cursor · grok-4.5

"Weak product fit as auto-summarize; a thin on-demand, cache-by-contentVersion experiment is fine. A confident wrong summary is worse than none."

arbiter · claude

Build the thin version. The AI binding, the invalidation key, and the access gate all already exist — the feature is mostly wiring, so the narrow value is worth the small cost.

02Unanimous — all three voices

Where all three reviews landed independently. Treat these as settled.

API worker only

Never the content worker — it is a deliberately thin, cookie-less serve path with no AI binding.

On-demand, never on-upload

Agents redeploy constantly; eager generation burns neurons on versions nobody reads. No waitUntil, no upload-path coupling.

contentVersion is the cache key

Already bumped on every replace and CAS-protected (upload.ts); rename/visibility PATCH doesn't touch it. Stored ≠ current → stale, never silently served as fresh.

HTMLRewriter extraction

Drop script/style, collect text, hard-cap ~8–16k chars. Reuse resolveIndexPath for the entry file; pure-audio sites short-circuit with zero AI calls.

Auth = checkAccess

Same gate as viewing the site (lib/access.ts, the single source of truth). Version derived server-side, never trusted from the client.

Mirror transcribeVoice

Pure lib/summarize.ts, never throws, degrades to "unavailable" without the binding — the established Workers-AI pattern in this repo.

03Splits — and the calls

Four genuine disagreements between codex and cursor. The highlighted column is the arbiter ruling.

Generation: Queues vs synchronous request
codexQueue-backed jobs with claim/lease semantics; waitUntil only guarantees 30s.
cursorSingle blocking POST; repo has no queue infra, don't invent one for v1.
ruling → synchronousOne R2 get + one AI call is a few seconds. Queues means a new binding, a consumer, and polling UX with no payoff. The 30s concern only applies to the on-upload path everyone already rejected.
Storage: columns on sites vs new table
cursorNullable summaryText / summaryForVersion columns on sites — joins with feeds.
codexDedicated site_summaries table with status, model, coverage metadata.
ruling → new tablesite_summaries (siteId unique, summary, contentVersion, model, generatedBy, truncated, createdAt). Keeps the hot sites row lean; feeds don't want summary text anyway.
Dashboard entry point: yes or viewer-only
cursorViewer-only; dashboard rows are too dense for AI blurbs.
codexOne reusable sheet, opened from a dashboard row action and the viewer top bar.
ruling → both, one sheetThe ask was per-site. A small sparkle row action opens the same sheet the viewer uses — density solved, no second surface built.
Who can regenerate
codexFirst summary: any viewer. Force-regen: canReplace users only (quota griefing).
cursorView-gate everything, add a rate limit; internal team.
ruling → any viewer + rate limitInternal team, no griefing surface. The canReplace gate would also require plumbing a capability the viewer payload doesn't expose today.
Round 2 · Regenerate while fresh? — "one summary per deployment"
cursorAlways allow — POST always regenerates; staleness is a signal, not a gate. A trapped garbage summary must not need a fake redeploy to fix.
codexFresh POST returns cached; explicit force:true regenerates. Model/prompt version joins the freshness check; generation lease for races.
ruling → one per deploy by default, quiet escapeServer-enforced: a fresh POST returns the cache with zero AI calls — idempotent and race-safe by construction. force:true hides behind a quiet "Regenerate anyway". Stale = contentVersion OR promptVersion mismatch, so a prompt fix surfaces "Update" everywhere without redeploys. A failed run is retryable and never overwrites a good summary. Codex's lease machinery rejected — overkill at team scale.
Round 4 · Azure OpenAI (company key) vs Workers AI
cursorAzure-only, gated Google-style (isSummaryEnabled); forks without a key simply lack the feature. Dual provider = two prompt/eval matrices, not worth it in v1.
codexConditional fallback: Azure gpt-4.1-mini when the three secrets exist, else Workers AI llama-70B, else unavailable. Never fail over mid-request. Matches the repo's optional-binding idiom.
ruling → Azure primary, Workers AI fallbackAll voices: Azure wins on quality and isolates quota from Whisper; Whisper stays put. Fallback kept (2–1): the prompt is identical text-in/text-out, the fallback model is what we'd have shipped anyway, and the self-hostable template gets Summarize at zero config via the already-committed AI binding. Provider resolved per attempt — an Azure error is a retryable failure, never a silent downgrade. Provider + model stamped on the row; freshness stays version + promptVersion.
Round 5 · "Ask AI" free-form questions — instead of, or with, the summary?
cursorSummary-only v1. Triage is the scarce need; Ask is ChatPDF-on-a-page-you're-reading, breaks the cache economics, and the ~12k entry-file extract can't honestly answer "where is X" on multi-file sites. Injection worse (question + content both untrusted).
codexSummary-only v1 — reviewer-question demand is a hypothesis with no evidence in the review loop. But design the generator as answerFromSite(task, context) so Summary IS the fixed-prompt case; instrument usage; add Ask later if demand shows.
ruling → summary v1, Ask designed-for but deferredBoth externals converge; arbiter's initial "both now" lean conceded. The generator ships as fixed-prompt Q&A internally, so Ask later = one ephemeral rate-limited route + a textarea in the same sheet (summary pinned top, no history, no answer cache, explicit "entry page only" scope). Trigger to build it: reviewers repeatedly asking the same questions a summary doesn't answer.

08Gotchas carried out of the consult

  • The committed wrangler.jsonc already binds AI ("ai": { "binding": "AI", "remote": true }) — both external agents false-flagged it missing because the local skip-worktree copy is stale. Verified against git HEAD.
  • Prompt injection (codex): uploaded HTML is untrusted prompt input. Low blast radius (no tools), mitigated with a hardened system prompt + a visible "AI-generated" label.
  • Neuron pool is shared with Whisper voice transcription — caching by version is what keeps this free-tier viable (~142 neurons per summary on the 70B model, ~70/day headroom).
  • Server never renders JS — extraction sees uploaded source only. Chart-heavy pages summarize thin; audio-only sites must short-circuit.
  • Keep the content worker out of it — anything imported into schema.ts ships in the content-worker bundle; the new table is fine, heavy imports are not.
  • Azure is a new content processor for internal HTML (codex) — confirm the deployment's data region/retention is company-approved before setting the secrets; presence of the secrets is the opt-in. Validate 4.1-mini vs 70B on real Glance pages during implementation before final commit.

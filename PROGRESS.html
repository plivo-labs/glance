The shape

3-agent review, reconciled

Type @ in a review comment → pick a user → on submit the client sends mentions:[userId]. The server (in the existing comment POST handlers, where the site + access are already resolved) intersects those ids against who can actually see the site, drops self, and fire-and-forget writes one notifications row each. The tagged user's homepage shows a bell badge + a dashboard inbox card, fed by the root loader; a click deep-links into the viewer with the thread open. Delivery is the router revalidator — refresh, or a single 60s tick. No WebSockets.

Scope calls (signed off): the viewer/review route is outside the app shell, so the bell lives on homepage/shell pages only — a notification click navigates into the viewer. Voice comments have no @ UI, so mentions are text-comment only in v1. Mentionable set mirrors checkAccess (incl. group shares), never a naive union.

04

Strategy

  1. Greenfield feature → most cases are new-module-spec (drive the new abstraction into existence), not red-now-bug. Per step: write the spec red → minimal impl → green. Don't batch.
  2. Seam tests land with their seam — S-A (migration+seed) before any P1/P2 case; S-B (share/member seeders) before the mentionable matrix; S-C before the fire-and-forget isolation case; S-D before the web pure helpers.
  3. The mentionable set is the one place to over-test: it's a property-equivalence pin against checkAccess across all four visibilities + both share kinds. Get C5–C7 green before wiring S5.
  4. Fire-and-forget (C15) is a correctness invariant, not a nicety: a mention-write blowing up must never 500 an otherwise-valid comment. Prove it with an injected throwing notifier.
  5. UI (P4) is mostly non-bun — extract the pure bits (href builder, token parser) to unit-test, and cover the wiring with one agent-browser smoke (C22) at the P4 gate.
  6. End of each phase: the thermo-nuclear gate → fix findings → full suite green → one commit scoped to the phase.
